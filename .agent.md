# 🤖 Contexto Global do Agente de IA (.agent.md)

Este arquivo serve como **Contexto Persistente de Sessão** para as IAs que desenvolvem a arquitetura do **CROM Pesquisador (p.crom.run)**.
Ao iniciar uma nova sessão, a IA deve ler este arquivo para entender o estado, a estrutura arquitetural e as regras de desenvolvimento sem precisar ler todos os arquivos isolados do repositório.

## 🎯 Sobre o Projeto
*   **Nome:** CROM Engine / CROM Pesquisador
*   **Domínio de Produção:** `p.crom.run`
*   **Stack Principal:** Go (Backend/Crawler), SQLite (Data), React+Vite+Tailwind (Frontend).
*   **VPS Info:** Hospedado numa máquina Linux, utilizando NGINX como proxy reverso. Porta do Frontend é compilada (estática no `/var/www/crom-pesquisador/frontend/dist/`), e Porta do Backend Go roda no `8098` via daemon.

## 🛠️ Arquitetura e Decisões de Design (NÃO ALTERAR SEM ORDEM)

1. **Singleton Connection Pool & Mente Autônoma:**
   *   O motor utiliza o SQLite em modo WAL (`PRAGMA journal_mode=WAL`) e `SetMaxOpenConns(1)` para evitar locks. O banco principal fica em `data/index/global_index.db`.
   *   A tabela `crawler_queue` armazena a fila da "Mente Autônoma" do crawler. Novos links descobertos são inseridos lá. Pesquisas do usuário no Frontend também injetam "Seeds de Busca" (ex: Wikipedia/DuckDuckGo) nesta fila.

2. **O Roteador da API e Knowledge Panels:**
   *   A API (`backend/api/api.go`) processa buscas FTS5 isoladas por tipo (Imagens, Vídeos, Acadêmico, Code) para evitar *starvation* de abas.
   *   **Oráculos (Knowledge Panels):** Intenções de busca explícitas (ex: Dólar, Bitcoin) invocam a Binance API em tempo real no Backend para popular um Painel Rico, ignorando limitações locais de resultados.

3. **Frontend (React) e Geo-Bloqueio:**
   *   Interface com Rolagem Infinita nativa (onScroll) via `SearchPage.tsx`. Abas segmentadas.
   *   **Defesa Regional:** Existe um `Geo-Block` diretamente no `App.tsx` usando `geojs.io`. Acesso fora do `BR` exibe a tela bloqueada (`BlockedRegionPage.tsx`) poupando latência da VPS.

4. **Crawler Engine (O Aracnídeo Fantasma):**
   *   O `crawler/main.go` agora possui um **Proxy Pool Rotativo** (`crawler/proxy.go`) que baixa IPs Brasileiros do ProxyScrape, impedindo bloqueio de firewall dos alvos.
   *   Roda em loop contínuo sem `config` (Modo Queue) extraindo sitemaps e varrendo a web via BFS (Breadth-First Search).

## 🔄 Procedimento Padrão de Atualização

Quando um usuário pede uma modificação de código durante uma sessão de Pair Programming, siga este fluxo:
1. **Análise de Camada:** Se pedir mudança na "Busca de Imagens", mexa em `SearchPage.tsx` e `api.go`. Se pedir "Ajuste na porta", mexa em `vps_sync.sh` e `api.go`.
2. **Edição do Código Local:** Utilize as ferramentas de edição no código local (Máquina do Usuário).
3. **Commit:** Execute um git commit utilizando mensagens no padrão Conventional Commits (ex: `fix(api): ...` ou `feat(ui): ...`) e faça `git push origin main`.
4. **Deploy de Produção:**
   Instrua o usuário a entrar na VPS e rodar o script oficial de sincrozinação: `./vps_sync.sh`

## ⚠️ Restrições Críticas (Segurança)

*   **Zero Hardcoding de Segredos:** Nunca salve senhas ou `ADMIN_SECRET` abertamente. Use o ambiente Linux (.bashrc / .env).
*   **Deletar Banco de Dados (NUNCA):** Sob nenhuma hipótese apague `global_index.db`, a menos que ordenado pelo usuário.
*   **Priorize o `grep_search` e `multi_replace_file_content`** para intervenções cirúrgicas. A base de código está viva.
