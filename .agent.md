# 🤖 Contexto Global do Agente de IA (.agent.md)

Este arquivo serve como **Contexto Persistente de Sessão** para as IAs que desenvolvem a arquitetura do **CROM Pesquisador (p.crom.run)**.
Ao iniciar uma nova sessão, a IA deve ler este arquivo para entender o estado, a estrutura arquitetural e as regras de desenvolvimento sem precisar ler todos os arquivos isolados do repositório.

## 🎯 Sobre o Projeto
*   **Nome:** CROM Engine / CROM Pesquisador
*   **Domínio de Produção:** `p.crom.run`
*   **Stack Principal:** Go (Backend/Crawler), SQLite (Data), React+Vite+Tailwind (Frontend).
*   **VPS Info:** Hospedado numa máquina Linux, utilizando NGINX como proxy reverso. Porta do Frontend é compilada (estática no `/var/www/crom-pesquisador/frontend/dist/`), e Porta do Backend Go roda no `8098` via daemon.

## 🛠️ Arquitetura e Decisões de Design (NÃO ALTERAR SEM ORDEM)

1. **Singleton Connection Pool (Banco de Dados):**
   *   Múltiplas goroutines (`Crawler`) gravam e a `API` lê dados de **um único arquivo** (`data/index/global_index.db`).
   *   O SQLite pode dar lock (`SQLITE_BUSY`). Para evitar isso, o arquivo `backend/db/db.go` usa um `sync.Once` e `SetMaxOpenConns(1)`. O SQLite está em WAL Mode (`PRAGMA journal_mode=WAL`).
   *   **Nunca:** Altere o pool de conexões sem extrema cautela. As requisições são serializadas na memória para não travar a I/O.

2. **O Roteador da API (Backend):**
   *   A rota de busca (`/search?q=XYZ`) no `api.go` não faz uma única query genérica. Para resolver o *starvation* de abas e a paginação, ele roda **múltiplas consultas FTS5** (`LIMIT 15` para Sites, 15 para Imagens, 15 para Vídeos).
   *   Cache de respostas é feito em memória via `sync.Map` com expiração de 5 minutos.
   *   `hasMore` é a booleana que controla o infinite scroll da UI.

3. **Scroll Infinito no Frontend:**
   *   O `SearchPage.tsx` abandonou carrosséis estilo Swiper. Usa-se um `div` normal com rolagem vertical. Ao chegar perto do final da `div`, o evento `onScroll` dispara `loadMore()` chamando a próxima página de forma fluida.

4. **Crawler Engine:**
   *   `crawler/main.go` e scripts secundários raspam HTML usando Colly, validam nós de mídias (imgs, iFrames de youtube, código `<pre>`) e invocam `IndexNode` no CRUD do backend.
   *   Roda em loop sem interrupções por um bash (`crawler_loop.sh`) ou via terminal pontual `--config`.

## 🔄 Procedimento Padrão de Atualização (O "Como Atualizamos")

Quando um usuário pede uma modificação de código durante uma sessão de Pair Programming, siga este fluxo:

1. **Análise de Camada:** Se pedir mudança na "Busca de Imagens", mexa em `SearchPage.tsx` e `api.go`. Se pedir "Ajuste na porta", mexa em `vps_sync.sh` e `api.go`.
2. **Edição do Código Local:** Utilize as ferramentas de edição no código local (Máquina do Usuário).
3. **Commit:** Execute um git commit utilizando mensagens no padrão Conventional Commits (ex: `fix(api): ...` ou `feat(ui): ...`) e faça `git push origin main`.
4. **Deploy de Produção:**
   Instrua o usuário a entrar na VPS e rodar o script oficial de sincrozinação:
   `./vps_sync.sh`
   *(Este script faz pull do github, compila o Frontend em Vite, compila o Backend Go e dá reload no Nginx).*

## ⚠️ Restrições Críticas (Segurança)

*   **Zero Hardcoding de Segredos:** Nunca salve senhas, `ADMIN_SECRET`, chaves SSH ou IP da VPS abertamente em arquivos do repo. Variáveis de ambiente sensíveis ficam ocultas no servidor Linux via profiles (`.bashrc` do root) ou em `.env` ignorados no git.
*   **Deletar Banco de Dados (NUNCA):** Sob nenhuma hipótese apague o arquivo `data/index/global_index.db` programaticamente, a menos que seja um comando explícito e destrutivo "WIPE" solicitado pelo usuário.
